## 0. Executive Summary

다운로드가 불가능한 대학 LMS(Manaba, WebClass 등) 강의 영상을 **재생만 하면** 실시간으로 음성과 화면을 분석해 타임스탬프 노트와 슬라이드 캡처를 생성하고, 강의 종료 후 PDF/Markdown으로 다운로드할 수 있게 해주는 Chrome 익스텐션. 일본 대학생을 1차 타겟으로 하며, 기존 YouTube 요약 도구나 업로드형 AI 학습 도구가 침투하지 못한 "보호된 강의 영상" 시장을 차지한다.

**핵심 차별점**: ① ダウンロード不可な動画で動作 ② リアルタイム ③ スライド認識 ④ 日本語ファースト

## 1. Goal & Target User

### 1.1 Problem Statement

일본 대학생, 특히 90분짜리 녹화 강의를 다수 시청해야 하는 학생은:

- 모든 영상을 처음부터 끝까지 듣기엔 시간이 부족함
- 강의는 LMS에 보호된 형태로 호스팅되어 **다운로드 불가**
- 따라서 NotebookLM 같은 업로드형 도구를 못 씀
- 기존 YouTube 요약 도구는 자막 API에 의존해서 LMS 영상엔 무용지물
- 노트를 손으로 정리하는 시간이 또 추가로 듦

### 1.2 Product Goal

**「再生するだけで、講義の要点が手元に残る。」**

사용자는 강의를 평소처럼 LMS에서 재생하기만 하면, 익스텐션이 음성과 화면을 실시간으로 분석해서:

1. **재생 중**: 사이드 패널에 타임스탬프 노트 + 슬라이드 캡처가 점진적으로 쌓임
2. **재생 후**: 사용자가 클릭해서 PDF / Markdown으로 노트 다운로드
3. **재방문**: 같은 URL에 다시 들어가면 기존 요약이 자동으로 사이드 패널에 표시

### 1.3 Target Persona (Primary)

- **이름**: タナカ ケンタ (가상)
- **소속**: 일본 사립/국립대 학부생 또는 대학원생
- **상황**: 학기당 6~10개 과목, 과목당 주 1~2회 90분 녹화 강의. 시험 기간엔 미시청 영상이 30~50개 누적
- **현재 행동**: 1.5~2배속 흘려듣기 + 손 필기. 이해 못 한 부분은 되돌려 보지만 시간 부족해서 그냥 넘김
- **욕구**: "30분 안에 강의 핵심 파악, 시험 전엔 노트만 보고 복습"

### 1.4 Differentiation One-liner

> **ダウンロード不可な講義動画専用 — Real-time + Slide-aware + Japanese-first**
> 다운로드 불가 강의 영상에서 동작하는 유일한 실시간 슬라이드 인지형 일본어 학습 어시스턴트.

## 2. User Flow

### 2.1 첫 사용 (Onboarding) — 1회성

1. Chrome 웹스토어에서 익스텐션 설치
2. 익스텐션 아이콘 클릭 → 환영 화면
3. Google SSO 또는 이메일 로그인 → 무료 플랜 자동 적용 (月 30분)
4. 권한 요청: 「現在のタブの音声・画面へのアクセス」
5. **면책사항 + 利用規約 동의 모달** (동의 없이 진행 불가)
6. 1분 데모 영상으로 튜토리얼 제공

### 2.2 코어 플로우 — 강의 시청

```
[STEP 0] 익스텐션이 ON 상태인지 확인
  → 툴바 아이콘 클릭 → 사이드 패널 열림 (계정 view)
  → 사이드 패널 헤더의 ON/OFF 스위치로 활성화 토글 (badge로 상태 표시)
  ↓
[STEP 1] 사용자가 LMS 강의 영상 페이지 접속
  ↓
[STEP 2] 익스텐션이 <video> 엘리먼트 자동 감지
  → 영상 우측 상단 모서리에 inline 버튼 표시 (📚 아이콘만)
  → 호버 시 「📚 この動画を要約」 텍스트 확장
  → 클릭 (수동 활성화)
  ↓
[STEP 3] In-page iframe 모달 자동 오픈 (400×600 floating, top-right 20px margin)
  - 페이지 위에 떠있는 floating 모달 (별도 Chrome 윈도우 아님)
  - 미인증 시: 모달 안에 LoginScreen → Google OAuth 로그인 가능 (모달 안에서 완결)
  - 인증 시: 즉시 세션 view (노트/슬라이드/정지/다운로드)
  - 헤더: [✕ 닫기]  (닫아도 세션은 백그라운드 지속, 인라인 버튼 클릭 시 재오픈)
  - Chrome 사이드 패널은 그대로 (계정 view) — 세션은 모달에서 별도로 진행
  ↓
[STEP 4] 익스텐션이 미리 설정된 배속으로 자동 전환
  → 디폴트: "영상 플레이어 제공 최고 배속" (자동 감지)
  → 사용자가 익스텐션 설정에서 임의 변경 가능
  ↓
[STEP 5] 사용자가 영상 재생 시작
  → 익스텐션이 captureStream() 후킹
  → 15초 단위 오디오 청크 → 백엔드 전송
  → 백엔드: STT → 슬라이딩 윈도우 LLM 요약
  ↓
[STEP 6] 사이드 패널에 5~10초 지연으로 노트가 한 줄씩 추가됨

  📝 ライブノート
  ┌────────────────────────────────────┐
  │ [00:42] AI の定義: 人間の知能を…    │ ← 클릭 시 영상 점프 + 1배속
  │ [02:15] 機械学習 vs 深層学習        │
  │ [05:30] ⭐ 重要: 誤差逆伝播法        │
  │ [08:14] 🖼️ スライド: ニューラル… │ ← 슬라이드 썸네일
  └────────────────────────────────────┘

  → 슬라이드 변경은 화면 픽셀 diff로 감지해 썸네일 첨부
  → 영상 옆 inline 버튼은 「処理中… ●」 status pill로 변환 (붉은 점 펄스)
  ↓
[STEP 7] 사용자가 핵심 줄 클릭 → 영상 점프 + 1배속 자동 전환
  → 정상속도로 청취 후 다시 빨리감기 모드 복귀
  ↓
[STEP 7-A] (선택) 사용자가 [⏹ 停止] 클릭 — 두 곳에서 가능
  → A. popup 풋터의 정지 버튼 (popup 열려 있을 때)
  → B. 인라인 버튼 status pill의 ⏹ 정지 버튼 (popup 안 열어도)
  → 오디오/슬라이드 캡처 즉시 정지
  → 누적된 노트는 보존 (popup 안에서 다운로드 가능; popup 닫혀 있으면 다음 인라인 클릭 시 재오픈하여 다운로드)
  → inline 버튼 status pill이 다시 「✨ 要約」 idle 상태로 복귀
```

### 2.3 종료 플로우 — 노트 저장 (수동)

```
[STEP 8] 영상 종료 또는 사용자가 [完了] 클릭
  → 사이드 패널에 [📥 ダウンロード] 버튼 표시 (자동 모달 X)
  ↓
[STEP 9] 사용자가 다운로드 버튼 클릭
  → 포맷 선택: PDF (슬라이드 첨부) / Markdown / 둘 다
  → 즉시 다운로드 시작
```

### 2.4 재방문 플로우 (Revisit)

```
[ENTER] 이전에 요약한 강의 URL 재접속
  ↓
[A] 익스텐션이 백그라운드에서 (user_id, normalized_url) 키로 서버 조회
  ↓
[B] 기존 요약 발견 → 사이드 패널 자동 슬라이드인
  - 저장된 노트 즉시 표시
  - 상단 배지: 「過去の要約: YYYY-MM-DD 作成」
  - 액션: [📥 다운로드] [🔄 추가 분석] [🗑️ 삭제]
  ↓
[C] 사용자 선택지:
  - 노트만 다시 보고 다운로드
  - [🔄 추가 분석]으로 새 부분 이어 처리 (Phase 2)
  - [🗑️ 삭제]
```

### 2.5 Edge Cases

| 시나리오 | 처리 |
|---------|------|
| 영상 일시정지 | 처리 일시정지, 재생 시 재개 |
| 사용자가 시킹(seek) | 노트 타임라인에 "스킵됨" 마커, 그 구간 분석 안 함 |
| 영상 도중 익스텐션/탭 전환 | 임시 저장, 복귀 시 이어서 |
| 무료 quota 소진 | "今月の無料枠を使い切りました" 모달 + 업그레이드 CTA |
| `<video>` 엘리먼트 없음 (canvas 기반) | "このサイトは現在対応していません" 안내 |
| 인터넷 연결 끊김 | 로컬 큐 적재, 복귀 시 일괄 전송 |

## 4. Technical Architecture

### 4.1 시스템 개요

┌──────────────────────────────────────────────────────────────┐

### 4.2 핵심 데이터 흐름 (재생 중)

1. Content Script: `videoElement.captureStream()` → MediaStream
2. 오디오 트랙 → AudioContext → 15s 청크 → Opus/WebM 인코딩
3. Service Worker: `POST /v1/stream/audio` (인증 헤더, 청크 idempotency key)
4. 백엔드: STT → 직전 N분 컨텍스트 + 신규 청크로 슬라이딩 윈도우 LLM 요약
5. WebSocket: `note_chunk` → Side Panel에 한 줄 추가
6. 비디오 프레임: 1fps 캔버스 샘플링 → 픽셀 diff → 임계치 초과 시 슬라이드 전환 판정 → JPEG 인코딩 → `POST /v1/stream/slide`
7. WebSocket: `slide_chunk` → Side Panel 노트에 썸네일 첨부
8. 영상 종료: `POST /v1/session/finalize` → 후처리 → DB 저장 → 다운로드 URL 반환

### 4.3 컴포넌트 책임

| 컴포넌트 | 책임 | 의존 |
|---------|-----|------|
| **Content Script** | DOM 접근, video 감지, MediaStream, 슬라이드 diff | Side Panel, Service Worker |
| **Side Panel (React)** | 노트 UI, WebSocket 수신, 다운로드 | Service Worker |
| **Service Worker** | 인증 토큰, REST/WS 클라이언트, 오프라인 큐 | Backend API |
| **API Gateway** | 인증 검증, quota, 라우팅 | Postgres |
| **Stream Processor** | 청크 수신/순서 보장, AI Pipeline 호출 | AI Pipeline, Storage |
| **AI Pipeline** | STT/LLM/OCR 호출 + 정규화 | External AI APIs |

### 4.4 핵심 엔지니어링 리스크 + 완화책

| 리스크 | 완화책 |
|--------|--------|
| iframe 안 video 접근 불가 | manifest `all_frames: true` + 호스트 권한, 일본 대표 LMS 3-4곳 사전 호환성 테스트 |

### 4.5 데이터베이스 선택 — 단계적 접근

**MVP / Closed Beta**: **RDS PostgreSQL `db.t3.micro` (single-AZ)**

- 단일 AZ — 데이터센터 장애 시 짧은 다운타임 가능 (closed beta 단계엔 acceptable)
- 7일 자동 백업

**Public Launch 직전 또는 트래픽 급증 신호 시**: **Aurora Serverless v2 + Multi-AZ**

- scale-to-zero (5분 idle 시 0 ACU) 적용 가능

- 동시 사용자 50명 초과

- 가용성 SLO < 99.5% 필요
- 피크 트래픽 ↑ 200% (시험 기간 등)

**마이그레이션 절차**: `docs/superpowers/specs/2026-04-26-aurora-migration-runbook.md` 참조 (read-replica promotion 방식, 다운타임 ~30초, 코드 변경 0).

## 10. Out of Scope (이 PRD에서 다루지 않음)

- 마케팅 / GTM 전략 상세
- 채용 / 조직 계획
- 자금조달 / 투자 유치
- 모바일 앱 (확정 제외)
- DRM 우회 (확정 제외)
- Safari / Firefox 버전 (Phase 후순위)
- **Cross-origin iframe 안 video 지원**: MVP 현재 버전은 top frame의 `<video>` 엘리먼트만 처리. iframe(예: 일부 LMS의 외부 video player 임베드) 안 video는 inline 버튼이 안 뜸. Phase 2의 P7 호환성 확장 작업에서 다룰 예정 |
- 드래그 가능 / 리사이즈 가능 모달: MVP 모달은 고정 위치(top-right 20px margin, 400×600). 사용자가 자유 이동 가능한 모달은 Phase 2+ 검토.
- **별도 Chrome 윈도우 popup (`chrome.windows.create`)**: 한 차례 시도했으나 macOS에서 fullscreen 렌더 + 사용자 의도와 다른 별도 윈도우라 폐기. In-page iframe 모달이 「브라우저 안 떠있는 모달」 사용자 의도와 일치하여 채택.
- **iframe full-height 사이드바 (페이지 우측 edge에 도킹)**: 콘텐츠 가림 + Chrome 네이티브 사이드 패널과 정보 중복으로 폐기. Floating 모달 형태(현재 채택)가 정답.
- **사이드 패널을 세션 view로 사용**: Chrome 116+ `chrome.sidePanel.open()` 은 직접적인 사용자 제스처(action 클릭/contextMenu 등)에서만 호출 가능. content-script-triggered 메시지 패스에서는 호출 불가. 따라서 사이드 패널은 계정 view 전용, popout 윈도우가 세션 view 전용이라는 역할 분리 적용 (Phase 2에서 Chrome 정책 완화되면 재검토)
