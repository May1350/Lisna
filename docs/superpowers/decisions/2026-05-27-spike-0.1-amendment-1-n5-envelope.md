# Spike 0.1 Amendment 1 — Hardware-reduced N=5 acceptance envelope

**Date**: 2026-05-27
**Status**: accepted
**Context**: Spike 0.1 (`zod-to-gbnf` HARD GATE) original Plan + spec acceptance was "10/10 LLM samples Zod-parse within ≤3 attempts." Three iter-1..3 runs at N=10 on the founder's M3-8GB dev machine produced kernel panics (×2 on 2026-05-26) under sustained Llama-3.2-3B-Q4_K_M inference. Take-3..6 at N=10 with the founder-selected Path 2 retry contract were also killed mid-suite by timeout / parent-shell exit / panic-avoidance. The trim commit `251c1fc` reduced PROMPTS 10→5 + added 5 s cooldown + `afterAll pkill` cleanup, and take-4 (commit `46ed08a`) cleared 5/5 within ≤2 attempts in 5.79 min wall. Take-5 (1B Q4_K_M co-validation) hit the same 5/5 profile at 2.39 min. But Plan §Spike 0.1 + spec §7.4 + decision memo title were not updated alongside, leaving the spec surface contradicting the running code (10/10 acceptance text vs PROMPTS.length=5 / expect(passCount).toBe(PROMPTS.length)).
**Decision**: Amend Plan §Spike 0.1 + Spec §7.4 + decision memo + `round-trip.test.ts` header to **"N/N within ≤3 attempts"** where **N = the largest sample count that fits the dev hardware's safe sustained-load envelope** — currently **N=5** on M3-8GB (pinned by `.claude/rules/pitfalls.md (spike-llm)`, commit `43d1f73`). Amendment STANDS until a founder commit explicitly raises N; migrating to ≥16 GB hardware does NOT auto-restore N=10. Full N=10 recovery procedures (Path 2.A foreground isolated rig / Path 2.B relocate to ≥16 GB machine / Path 2.C combine with Path 1 bounded-array grammar) live in `desktop/spikes/phase-0/01-zod-to-gbnf/decision-0.1-fail.md` "Path 2.A/B/C procedures" subsection.
**Alternatives considered**:
- **Lower hardware target permanently** — rejected. Alpha spec is `Apple Silicon ≥ 16 GB`; the M3-8GB constraint is dev-time, not production.
- **Switch capability floor to 1B model entirely** — rejected as HARD GATE substitute. 1B PASSed 5/5 (take-5) but quality-floor (slot emergence on real Lecture content) is unproven; Spike 0.2's 1B path is queued as Path F separately.
- **Abandon 3B and adopt 7B on ≥16 GB only** — rejected. 7B is roughly 2× slower per token (Spike 0.2 Path E generation budget already 28 s at 3B); breaks the on-device latency envelope and abandons capability-floor work.
- **Accept PARTIAL as PASS without spec amendment** — rejected as anti-option. HARD GATE forbids "n/10 + probably fine on the rest"; preserving the spec-evidence contradiction would mislead every future reader and weaken HARD GATE discipline for future spikes.
- **Path 1 (bounded-array grammar) as the primary fix** — deferred (not rejected). Structurally eliminates failure mode A but doesn't address mode B (string char-escape loop). Listed as Path 2.C combo; effort 2-3 h vs the retry contract's already-shipped status.
**Consequences**:
- Spike 0.2 (recipe selection) green-lit at N=5 reduced scope (verdict memo `44e546d`); Spike 0.2 itself returned MIXED on latency, separately tracked.
- Production retry budget (`maxAttempts=3`, fresh seed `+100×(attempt-1)`, JSON+Zod catch→retry, log `attemptsUsed`/`reason`) becomes **load-bearing, not optional**. The retry contract already recovered i=0 photosynthesis at iter-4 temp=0.6 in both take-4 and take-6, and is the only mechanical defense against mode-A and mode-B in production until Path 2.A/B/C lands. Plan 2 (Foundation) wrapper must enforce via failing test.
- i=8 Maxwell (iter-3 failure-mode-B sample, original 10-prompt index 9) remains the empirically unverified production risk. Worst-case if a user generates formula-heavy content (Maxwell-class) and all 3 retries hit mode B: up to 24 min UI hang at 4096-token saturation × 3. Mitigation: per-attempt wall-time cap (90-120 s) + UI retry counter (queued as P2 in REFACTOR_BACKLOG).
- Spec amendment-in-same-commit pair-update discipline now captured in `.claude/rules/pitfalls.md (test-headers)` to prevent future silent-trim drift.
