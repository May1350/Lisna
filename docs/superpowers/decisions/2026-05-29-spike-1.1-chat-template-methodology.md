# Spike 1.1 — apply the model chat template to the merge spike

**Date**: 2026-05-29
**Status**: accepted

**Context**: Spike 1.1 (Plan 6 Phase B) tests the interview merge-LLM call. It
was built to send a RAW combined prompt (no chat template) as a deliberate
"conservative lower bound." On re-run, that raw mode pushed the 3B instruct
model out-of-distribution: it failed to emit EOS (`<|eot_id|>`) and ran on until
`maxTokens` truncated the JSON mid-string. The prior session misdiagnosed the
resulting `JSON.parse` "Bad control character" as a GBNF char-rule gap (finding
F1) and patched the grammar, which could not fix it. (A separate harness UTF-8
decode bug, fixed in `73fb679`, was corrupting JA output in parallel.)

**Decision**: Apply the model's embedded Llama-3.2 chat template via `--jinja`
(systemTemplate → `-sys`, userPrompt → `-p`), mirroring the production sidecar,
instead of the raw-prompt lower-bound mode. (`8b9ca96`.)

**Alternatives considered**:
- Bound string lengths in the grammar (`char{0,N}`, Path G) — rejected: blunt
  (truncates legitimate content), less production-faithful, Path G unlanded.
- Record the finding and keep raw mode — rejected: yields no merge data, so the
  spike's core question (merge quality) stays unanswered.

**Consequences**:
- The spike now measures the production path, so the verdict is
  production-representative, not a pessimistic lower bound. With templating,
  chunk 1 passes and the merge runs → MIXED verdict obtained.
- Small rig change: `chatTemplate`/`systemPrompt` opt-in on `RunLlamaOptions`;
  phase-0 raw mode unchanged. The rig also strips llama-completion's
  `> EOF by user` epilogue before parse so truncation reports honestly.
- Finding F1 (GBNF char rule) is re-classified blocking → low-priority hardening
  (its evidence was the truncation, not a grammar gap); the spawn_task'd Plan-2
  chip should be re-scoped accordingly.
