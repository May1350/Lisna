# i18n rules — Lisna web marketing site

The marketing site (`web/`) ships in **EN / JA / KO**. All three must
stay in lockstep — a single hardcoded string or unsynced JSON key shows
up as broken language switching for end users.

Open this file before:
- Editing any `.tsx` under `web/src/app/[locale]/` or `web/src/components/`
- Editing `web/src/messages/{en,ja,ko}.json`
- Adding a new marketing surface, signin step, legal page, or pricing tier
- Adding a new locale to `web/src/i18n/routing.ts`

Spec captured 2026-05-24 (PR establishing i18n parity infrastructure).

## Architecture

| File | Role |
|---|---|
| `web/src/messages/{en,ja,ko}.json` | Per-locale message catalogs. Key structure MUST be identical across all three. |
| `web/src/i18n/routing.ts` | Locale list, default, prefix mode. `localePrefix: 'as-needed'` — `/` is `en`, `/ja/...` is JA, `/ko/...` is KO. |
| `web/src/i18n/request.ts` | next-intl loader; falls back to default locale on invalid input. |
| `web/src/i18n/navigation.ts` | Wrapped `Link` helper. Partial migration — only legal pages use it; rest still rely on manual `prefix = locale === 'en' ? '' : '/${locale}'`. |
| `web/src/i18n/brand-vocabulary.ts` | **Single source of truth** for strings that NEVER translate (product names, model names, license codes, competitor names, currency symbols) + allowlists for the check script. |
| `web/scripts/check-i18n.mjs` | Verifier — run in pre-commit (warn) + CI (strict). |
| `.claude/skills/i18n-check/SKILL.md` | Claude-side workflow for resolving violations. |

## The hard contract

- [2026-05-24] (parity) Every key in `messages/en.json` MUST exist in `ja.json` AND `ko.json`. No stray keys allowed in either direction. Pre-commit + CI enforce this; CI runs in `--strict` mode. Reason: missing keys render as the raw key path (e.g. `pricingSection.heading`) in production. last-cited: 2026-05-24
- [2026-05-24] (parity) If a JA or KO value is byte-identical to the EN value, the key MUST be allowlisted in `IDENTICAL_VALUES_OK` (with a one-line WHY comment). Otherwise it's flagged as "copy-paste, never translated." Reason: the common failure mode for partial translation passes. last-cited: 2026-05-24
- [2026-05-24] (brand) Strings that should never translate (`Lisna`, `Whisper`, `Llama`, `Obsidian`, `Apple Silicon`, competitor names like `Otter` / `Fireflies` / `Notion AI`, currency `¥` / `$`, license codes like `MIT` / `Meta license`, `Discord` / `GitHub` / `Bluesky`) MUST be imported from `BRAND` in `brand-vocabulary.ts`. Don't hardcode them in JSX, don't put them in `messages/*.json`. Reason: renaming a brand should touch one file, not 47. last-cited: 2026-05-24
- [2026-05-24] (cjk) No hiragana / katakana / kanji / hangul characters allowed in `.tsx` under `web/src/app/` or `web/src/components/`, except: files allowlisted in `HARDCODED_OK_FILES` (tokusho, hero decorative demo, Stripe callback pages, locale-switcher test); `<article lang="ja">` blocks in legal pages (paired with `<article lang="en">` in the same file); per-locale lookup objects (`{ en: '...', ja: '...', ko: '...' }`); `LOCALE_LABELS` / `LOCALE_SHORT` references. Reason: catches the regression where a developer pastes a Japanese string into JSX bypassing the translation flow. last-cited: 2026-05-24
- [2026-05-26] (hardcoded-en) `check-i18n` catches hardcoded CJK but NOT hardcoded ENGLISH (intentional — flagging every English literal needs an AST, too many false positives). So a `'use client'` component with baked-in English UI text (e.g. `EmailMagicLinkForm`'s `'Send link'` / `'Sending…'` / sent-confirmation) renders untranslated on `/ja` + `/ko` and still passes the check. Pattern to follow: presentational client components take their visible strings as PROPS with English defaults (keeps the sandbox + unit tests working without an i18n provider); the server/`'use client'` wrapper that has `t = useTranslations(...)` supplies the localized values (`submitLabel={t('sendLink')}` …). See `email-magic-link-form.tsx` (dumb, prop-driven) + `sign-in-panel.tsx` (wires `t(...)`). Reason: a default-English label looks fine in EN review and only surfaces as untranslated text on the localized routes. last-cited: 2026-05-26

## When you add UI copy

1. Write the EN version in `messages/en.json` under the right namespace.
2. Translate to JA and KO immediately. Don't copy EN as a placeholder; pass-2 of `check-i18n` will flag it the next time someone runs it.
3. In JSX, render via `useTranslations` (client) or `getTranslations` (server). NEVER inline the string.
4. If the rendered text contains a brand token (Lisna, Whisper, etc.), reference `BRAND.appName` etc. from `@/i18n/brand-vocabulary` rather than baking it into the message string. Exception: when the brand is INSIDE running prose ("Lisna runs on your Mac"), the brand is part of the message string and the prose around it gets translated. Use judgement — if removing the brand from the string would break grammar in one of JA/KO, leave it in the message.
5. Run `pnpm --filter lisna-web check:i18n` before opening the PR.

## When you add a new locale

1. Add the code to `web/src/i18n/routing.ts` `locales`.
2. Create `web/src/messages/<code>.json` — start by copying `en.json` then translating.
3. Add the entry to `LOCALE_LABELS` + `LOCALE_SHORT` in `brand-vocabulary.ts`.
4. Add the entry to `LOCALES` in `web/scripts/check-i18n.mjs`.
5. If any legal pages should serve the new locale's content rather than falling back to EN, update `ENGLISH_LOCALES` in `web/src/app/[locale]/{privacy,terms,refunds}/page.tsx` accordingly (today, `ko` falls through to EN; `tokusho` stays JA-only for all locales).
6. Run `pnpm --filter lisna-web check:i18n:strict`.

## Legal pages — special case

Privacy / Terms / Refunds each carry BOTH a JA and an EN block in the
same `.tsx` file. The page chooses via `ENGLISH_LOCALES`:
- `ja` → JA block (`<article lang="ja">`)
- `en` + `ko` → EN block (`<article lang="en">`) — Korean translation deferred pending legal review.
- Tokusho (`特定商取引法に基づく表記`) is a Japan-specific legal requirement and stays JA-only regardless of route.

When refreshing legal copy, edit BOTH blocks. The check script skips the
JA-tagged block on the assumption that it's intentional; the EN block is
the one subject to value-parity checks.
